# AI Agent с многоуровневой памятью и стратегиями контекста

Проект представляет собой интеллектуального агента, использующего Yandex GPT API, с поддержкой трёх слоёв памяти и трёх стратегий управления контекстом. Агент предназначен для персонализированного диалога с учётом профиля пользователя, текущей задачи и истории взаимодействия.

## Архитектура

### Три слоя памяти
1. **Краткосрочная память (ShortTermMemory)** – хранит последние N сообщений диалога (скользящее окно).
2. **Рабочая память (WorkingMemory)** – структурированные данные текущей задачи (ключ-значение).
3. **Долговременная память (LongTermMemory)** – профиль пользователя (роль, уровень экспертизы, предпочтения) и важные факты, извлечённые из диалога.

### Три стратегии управления контекстом
1. **Sliding Window** – классическое скользящее окно последних сообщений.
2. **Sticky Facts** – автоматически извлекает и сохраняет важные факты из диалога, добавляя их в системный промпт.
3. **Branching** – позволяет создавать параллельные ветки диалога и переключаться между ними.

### Состояние задачи как конечный автомат
Агент теперь включает формализованное состояние задачи, реализованное как конечный автомат (Finite State Machine). Это позволяет отслеживать прогресс задачи, управлять этапами и обеспечивать возможность паузы/продолжения без потери контекста.

**Ключевые компоненты:**
- **Этапы (stages):** planning → execution → validation → done (а также paused, error)
- **Текущий шаг:** текстовое описание текущего действия внутри этапа
- **Ожидаемое действие:** что пользователь или система должны сделать дальше
- **История шагов:** запись всех выполненных шагов с результатами
- **Метаданные:** дополнительные данные задачи, сохраняемые между этапами

**Возможности:**
- **Пауза на любом этапе:** задача может быть приостановлена и возобновлена позже
- **Продолжение без повторных объяснений:** состояние автоматически восстанавливается из сохранённых данных
- **Валидация переходов:** система проверяет допустимость перехода между этапами
- **Сериализация:** состояние задачи сохраняется в файл вместе с остальной памятью агента

**Новые команды в интерфейсе агента:**
- `/task_state` – показать текущее состояние задачи
- `/task_stage <stage>` – перейти на указанный этап (planning, execution, validation, done, paused, error)
- `/task_pause [причина]` – поставить задачу на паузу
- `/task_resume [stage]` – возобновить задачу (опционально указать этап)
- `/task_step <описание>` – обновить текущий шаг
- `/task_complete <результат>` – завершить текущий шаг и записать результат

## Установка

1. Клонируйте репозиторий:
   ```bash
   git clone <repository-url>
   cd AI_Agent
   ```

2. Создайте виртуальное окружение (рекомендуется):
   ```bash
   python -m venv .venv
   .venv\Scripts\activate   # Windows
   source .venv/bin/activate # Linux/Mac
   ```

3. Установите зависимости:
   ```bash
   pip install -r requirements.txt
   ```

4. Создайте файл `.env` в корне проекта со следующими переменными:
   ```
   YANDEX_CLOUD_FOLDER=your_folder_id
   YANDEX_CLOUD_API_KEY=your_api_key
   YANDEX_CLOUD_MODEL=yandexgpt/latest   # опционально, по умолчанию yandexgpt/latest
   ```

## Система инвариантов

Агент теперь включает систему инвариантов — правил, которые он не имеет права нарушать. Инварианты хранятся отдельно от диалога и явно учитываются в рассуждениях. Агент отказывается предлагать решения, которые нарушают инварианты, и объясняет причину отказа.

### Категории инвариантов
1. **Архитектура (architecture)** – выбранная архитектура системы (трёхслойная память, стратегии).
2. **Технические решения (technical_decisions)** – принятые технические решения (использование Yandex Cloud API).
3. **Ограничения по стеку (stack_constraints)** – ограничения по технологическому стеку (Python 3.9+, запрет сторонних библиотек).
4. **Бизнес-правила (business_rules)** – бизнес-ограничения (специализация на цифровых двойниках промышленных объектов).
5. **Прочие (other)** – любые другие правила.

### Как это работает
- Инварианты загружаются из файла `data/invariants.json` при старте агента.
- Каждый инвариант содержит ID, категорию, описание, обоснование и уровень серьёзности.
- Перед обработкой пользовательского запроса агент проверяет, не нарушает ли запрос инварианты (простые ключевые слова и паттерны).
- Если нарушение обнаружено, агент возвращает объяснение и отказывается выполнять запрос.
- После генерации ответа агент также проверяет собственный ответ на нарушение инвариантов (пост-проверка).
- Инварианты включаются в системный промпт, чтобы модель явно учитывала их при генерации.

### Новые команды
- `/invariants` – показать все активные инварианты.
- `/add_invariant категория "описание" ["обоснование"]` – добавить новый инвариант.
- `/remove_invariant id` – удалить инвариант по ID.
- `/check_invariants текст` – проверить произвольный текст на нарушение инвариантов.

### Пример конфликта
```
Вы: Давайте перейдем с Python на JavaScript и используем OpenAI вместо Yandex Cloud.
🤖 Агент: ⚠️ Запрос нарушает следующие инварианты:
  • Используется Yandex Cloud LLM API, а не OpenAI или другие провайдеры
    Нарушение: Предлагается использовать другого провайдера LLM вместо Yandex Cloud
    Категория: technical_decisions
    Обоснование: Решение принято из-за требований к локализации и стоимости
  • Проект написан на Python 3.9+, без использования других языков
    Нарушение: Предлагается использовать другой язык программирования вместо Python
    Категория: stack_constraints
    Обоснование: Ограничение стека для упрощения поддержки
```

### Реализация
- Модуль `agent/invariants.py` содержит классы `Invariant`, `InvariantCategory`, `InvariantManager`.
- `InvariantManager` отвечает за загрузку, сохранение, проверку и форматирование инвариантов.
- Интеграция в `Agent.ask()`: проверка до и после генерации ответа.
- Инварианты по умолчанию определяются в коде и автоматически создаются при первом запуске.

## Детали реализации

### Класс Agent (`agent/agent.py`)
- Инициализирует клиент Yandex GPT API через библиотеку `openai` с кастомным `base_url`.
- Управляет тремя слоями памяти и стратегиями.
- Сохраняет и загружает состояние в JSON‑файл (`data/agent_state.json`).
- Подсчитывает использованные токены.

### Стратегии (`agent/strategies.py`)
- **SlidingWindowStrategy** – передаёт в промпт последние `window_size` сообщений.
- **StickyFactsStrategy** – после каждого ответа вызывает модель для извлечения фактов и сохраняет их.
- **BranchingStrategy** – позволяет создавать независимые ветки диалога, каждая со своей историей.

### Память (`agent/memory.py`)
- **ShortTermMemory** – очередь сообщений с ограничением по длине.
- **WorkingMemory** – словарь для данных текущей задачи.
- **LongTermMemory** – хранит профиль пользователя и список фактов с временными метками.

## Возможные улучшения

1. **Добавление новых стратегий** – например, стратегии, учитывающей семантическое сходство для поиска релевантных фактов.
2. **Визуальный интерфейс** – веб‑интерфейс на Streamlit или Gradio.
3. **Интеграция с векторными БД** – для долговременной памяти вместо простого списка фактов.
4. **Поддержка других провайдеров LLM** – OpenAI, Anthropic, локальные модели.
5. **Расширенное логирование** – запись диалогов и метрик в базу данных.
